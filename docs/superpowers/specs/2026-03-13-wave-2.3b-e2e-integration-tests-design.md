# Wave 2.3b: E2E Integration Tests — Design Spec

> **Date**: 2026-03-13
> **Status**: Draft
> **Scope**: Integration tests for OpenClaw + Sentinel pipeline (Wave 2.3a code)
> **Prerequisite**: Wave 2.3a complete (48 unit tests, plugin package, executor endpoints, delegation system)

---

## Goal

Validate that Wave 2.3a components work together as an integrated pipeline. The 48 existing unit tests cover individual modules; this wave covers the seams between them.

**Non-goals**:
- Tool recursion depth limiting (Wave 2.3c)
- Context budget enforcement (Wave 2.3c)
- Multi-process or Docker-based E2E tests
- Real OpenClaw Gateway or Claude Code spawning

---

## Approach

**Single test file**: `packages/executor/src/integration.test.ts`

Follows the existing `smoke.test.ts` pattern:
- Real `createApp()` with real `AuditLogger`, `ToolRegistry`, `DelegationQueue`
- Temp directory for SQLite databases
- Hono `app.request()` for HTTP calls (no real server)
- Plugin tested both in-process (sanitizeOutput, health monitor) and via HTTP (classify, filter-output)
- No new dependencies

### Test Setup

```typescript
beforeEach(() => {
  tempDir = realpathSync(mkdtempSync(join(tmpdir(), "sentinel-integration-")));
  auditLogger = new AuditLogger(join(tempDir, "audit.db"));
  registry = createToolRegistry();
  delegationQueue = new DelegationQueue(join(tempDir, "delegation.db"));
  app = createApp(config, auditLogger, registry, undefined, undefined, delegationQueue);
});

afterEach(() => {
  auditLogger.close();
  delegationQueue.close();
  rmSync(tempDir, { recursive: true, force: true });
});
```

## Test Cases (8 tests)

### 1. Plugin classify → auto_approve

**Flow**: POST `/classify` with a read tool → auto_approve decision

```
POST /classify { tool: "read_file", params: { path: "/tmp/test" }, agentId: "oc-agent", sessionId: "s1", source: "openclaw" }
→ { decision: "auto_approve", category: "read", reason: "...", manifestId: "uuid" }
```

**Asserts**:
- Response status 200
- `decision === "auto_approve"`
- `category === "read"`
- Audit entry exists with `source: "openclaw"`

### 2. Plugin classify → block

**Flow**: POST `/classify` with a dangerous interpreter command → block decision

```
POST /classify { tool: "bash", params: { command: "python3 -c 'print(1)'" }, agentId: "oc-agent", sessionId: "s1" }
→ { decision: "block", category: "dangerous", reason: "..." }
```

**Asserts**:
- Response status **403** (block returns 403, not 200)
- `decision === "block"`
- `category === "dangerous"`

### 3. Plugin classify → confirm → execute

**Flow**: Classify returns `confirm` for a write tool, then execute with confirmation approval.

```
Step 1: POST /classify { tool: "write_file", params: { path: "/tmp/test", content: "hello" }, agentId: "oc-agent", sessionId: "s1" }
→ { decision: "confirm", category: "write" }

Step 2: POST /execute with full ActionManifest { id: <pre-generated UUID>, tool, parameters, agentId, sessionId }
→ (pending confirmation — uses manifest.id for confirmation routing)

Step 3: POST /confirm/<manifest.id> { approved: true }
→ { status: "approved" }

Step 4: Execute resolves with success
```

**Asserts**:
- Classify returns `confirm`
- Execute completes after confirmation
- Audit entry records the full lifecycle

### 4. Filter output pipeline

**Flow**: POST `/filter-output` with text containing both API keys and PII.

```
POST /filter-output {
  output: "API key is <dynamically-constructed-key> and SSN is 123-45-6789",
  agentId: "oc-agent"
}
→ { filtered: "API key is [REDACTED] and SSN is [PII_REDACTED]", redacted: true, moderationFlagged: false, moderationBlocked: false }
```

**Asserts**:
- `filtered` contains no API key pattern
- `filtered` contains no SSN pattern
- `redacted === true`
- Both credential and PII redaction applied in single pass

### 5. Health monitor fail-closed

**Flow**: Create plugin with unreachable executor URL → beforeToolCall blocks.

```typescript
const plugin = createSentinelPlugin({
  executorUrl: "http://127.0.0.1:19999", // nothing listening
  failMode: "closed",
  healthCheckIntervalMs: 100,
});
// Wait for health check to fail
await sleep(250);

const result = await plugin.beforeToolCall({
  toolName: "bash",
  params: { command: "echo hi" },
  runId: "run-1",
  session: { agentId: "oc-agent", sessionId: "s1" },
});
→ { block: true, blockReason: "Sentinel executor unreachable — fail-closed mode" }
```

**Asserts**:
- `block === true`
- `blockReason` mentions unreachable/fail-closed
- Plugin cleaned up via `plugin.stop()`

### 6. Delegation E2E

**Flow**: Delegation lifecycle via direct DelegationQueue + executor endpoints. Note: `delegate.code` is NOT registered in `createToolRegistry()`, so we test the delegation endpoints directly by pre-populating the queue.

```
Step 1: delegationQueue.enqueue({ id, task, status: "pending", ... }) — direct queue insertion

Step 2: GET /pending-delegations
→ [{ id, task (redacted), status: "pending", ... }]

Step 3: POST /delegation-status/:id { status: "completed", prUrl: "https://github.com/org/repo/pull/1" }
→ { ok: true }

Step 4: GET /pending-delegations → [] (no longer pending)
```

**Asserts**:
- Delegation appears in pending list after enqueue
- Task string is redacted (credential patterns stripped)
- Status update succeeds with valid Zod body
- Delegation removed from pending after completion
- Audit entry exists for delegation status update

**Note**: Production `entrypoint.ts` does not yet wire `delegationQueue` into `createApp()` — this is a known gap to address when delegation goes live.

### 7. Shared audit sources

**Flow**: Multiple requests with different `source` values, verify audit DB contains both.

```
Step 1: POST /classify { ..., source: "openclaw" }
Step 2: POST /execute { ... } (default sentinel source)
Step 3: Query audit DB directly
→ Entries with source="openclaw" AND source=undefined (sentinel default) both present
```

**Asserts**:
- At least one entry with `source === "openclaw"`
- At least one entry with `source === undefined` (sentinel default — `rowToEntry` maps SQL NULL to `undefined`)
- Both entries have valid Merkle chain hashes

### 8. sanitizeOutput direct import path

**Flow**: Test the plugin's synchronous sanitization (no HTTP).

```typescript
const plugin = createSentinelPlugin({ executorUrl: "http://localhost:3141" });
const result = plugin.sanitizeOutput(
  "key: <dynamically-constructed-api-key> and email: john@example.com SSN: 123-45-6789"
);
plugin.stop();
```

**Asserts**:
- No API key pattern in output
- No SSN pattern in output
- Email redacted (PII)
- Function is synchronous (returns string, not Promise)

## Risks

1. **Health monitor test timing**: Test #5 creates a plugin pointed at a dead port. Real TCP connections can be flaky (timeout vs connection refused). Mitigate by mocking `globalThis.fetch` to reject immediately, or use a `node:http` server that returns 500. `vi.useFakeTimers()` alone won't help since `fetch()` is the real bottleneck.
2. **Delegation flow simplified**: Test #6 pre-populates the queue directly instead of going through `/execute`, since `delegate.code` is not in `createToolRegistry()`. This tests the endpoint integration without requiring full tool execution.
3. **API key test fixtures**: Fake API keys may trigger GitHub push protection. Use dynamic construction (join array segments) per MEMORY.md pattern.
4. **Production delegation wiring gap**: `entrypoint.ts` doesn't pass `delegationQueue` to `createApp()`. Tests mask this by passing it directly. Track as known gap.
